# UXセルフレビュー・チェックリスト

自動開発エージェントによる自己点検の記録。v0.2時点（localStorageデモモード）の
医師側・病院側の全画面遷移を`index.html`・`js/app.js`・`js/store.js`・`js/master.js`・
`css/app.css`を読んで実際に追い、UX上の問題点を洗い出した。
`docs/SECURITY_CHECKLIST.md`と同様、**修正済み／未修正を明記**する。

## 1. 危険な操作の確認ダイアログ

| 操作 | 問題 | 対応 |
|---|---|---|
| 病院が応募者を「承認」（`doApprove`） | 承認すると同じ枠の他の応募者が自動的にお断り（`api.approve`内で自動declineされる）になるが、その副作用を病院側に一切伝えずワンクリックで実行されていた。 | ✅修正：`confirm()`で「承認すると、この枠の他の応募者は自動的にお断りになります。」と明示してから実行するように変更（`js/app.js` `doApprove`）。 |
| 病院が応募者を「お断り」（`doDecline`） | 確認なしの一発実行。取り消し手段もない。 | ✅修正：`confirm()`で意思確認を追加。 |
| 病院が「勤務完了」（`doComplete`） | 確認なしの一発実行。完了後は医師の実績カウントが増え、取り消し不可。 | ✅修正：`confirm()`で意思確認を追加。 |
| 医師の「取り下げ」（`doWithdraw`） | `prompt()`で理由入力＋キャンセル可能なので実質的に確認ダイアログを兼ねている。 | 未修正（現状で実質確認になっており緊急度低）。ただしネイティブ`prompt()`はアプリのモーダルUIと見た目が異なる。今後モーダル化するなら`openModal`ベースに統一するのが望ましい（今回は見た目変更のみで機能に影響しないため、小規模リファクタの範囲を超えるとみなし見送り）。 |
| 運営の「実在確認・却下」「デモデータ初期化」 | 却下には確認なし／初期化には`confirm()`あり。却下は管理者のみが到達する画面で誤操作リスクが低いため今回は対象外。 | 未修正（記録のみ）。 |

## 2. 入力ミスによる予期しない挙動

| 問題 | 対応 |
|---|---|
| 募集作成ウィザードの「報酬（円）」で未入力・非数値のまま「次へ／公開する」を押すと`+wiz.pay||0`で**無言のうちに¥0の募集が公開**されてしまう可能性があった（日にちのみバリデーションがあり、報酬は未検証だった）。 | ✅修正：`wizNext()`にステップ2（報酬入力）でのバリデーションを追加し、`wizPublish()`本体にも日にち・報酬の防御的チェックを追加。未入力時はトーストで警告し該当ステップへ戻す（`js/app.js`）。 |
| チャット入力欄で、日本語入力（IME）変換確定のEnterキーが誤って「送信」に解釈され、変換途中のメッセージが送信されてしまう不具合があった（`onkeydown="if(event.key==='Enter')sendChat()"`はIME合成中のEnterを区別していなかった）。 | ✅修正：`event.isComposing`および`keyCode===229`を除外する条件を追加（`js/app.js` `drawChat`のチャット入力欄）。 |
| ログイン・新規登録ボタンを連打すると、`await`中に多重に`auth.login`/`auth.signup`が走る可能性があった（多重送信・監査ログの重複記録リスク）。 | ✅修正：非同期処理中はボタンを`disabled`にし、完了後に再度有効化（`js/app.js` `doLogin`/`doSignup`）。CSSに`.btn:disabled`のスタイルを追加（`css/app.css`）。 |

## 3. 必須入力の分かりやすさ

| 問題 | 対応 |
|---|---|
| 医師プロフィール登録・病院情報登録フォームで、どの項目が必須でどれが任意か、ラベルを読むまで分からなかった（一部は「（任意）」と明記されているが、必須項目には印がなかった）。 | ✅修正：氏名・医籍登録番号・診療科・対応業務・免許証画像・本人確認書類（医師側）、病院名・住所（病院側）のラベルに赤字「＊必須」を追加。病院側の「代表電話」は実装上任意項目のため「（任意）」を明記。 |
| フォームの`<label>`が`<input>`と`for`/`id`で紐付いておらず、スクリーンリーダーでラベルとフィールドの対応が読み上げられない、クリックでフォーカスできない等の軽微なアクセシビリティ課題があった（ログイン・新規登録・医師/病院オンボーディング全フォーム）。 | ✅修正：該当する`<label>`に`for`属性を追加し対応する入力の`id`と紐付け。診療科・対応業務のチェック群（ボタン群でネイティブフォーム部品でないもの）は`role="group"`＋`aria-labelledby`で補足。 |
| バリデーションエラーの表示位置がトースト（画面下部・2.6秒で自動消滅）のみで、どの入力欄が原因か分かりにくい。 | 未修正。エラー該当フィールドの枠を赤くする等は`app.js`のレンダリング構造（フォームを都度`innerHTML`で再構築）に手を入れる必要があり、小規模な修正の範囲を超えると判断し見送り。今回は「＊必須」表示の追加で入力前の予防を強化するに留めた。 |

## 4. その他確認した観点（問題なし、または対応見送り）

- **エラーメッセージ全般**：`store.js`の全API関数は`{err:"..."}`形式で失敗理由を返し、`app.js`側は必ず`toast("⚠️ "+r.err)`で表示していた。表示されないケースは見当たらなかった。
- **空状態の表示**：募集なし／応募なし／確認待ちなしのケースはすべて日本語の案内文が用意されていた（例：「現在、公開中の募集はありません」「まだ手上げがありません。登録医師に公開中です。」）。問題なし。
- **ローディング表示**：地図読み込み失敗時のフォールバック文言はあり。ログイン/登録のようなハッシュ計算（`crypto.subtle.digest`）を伴う処理はミリ秒単位で完了するため専用スピナーは不要と判断し、連打防止（上記2.）のみ対応。
- **画面遷移の分かりやすさ**：募集詳細に「‹ 一覧に戻る」の戻り導線あり。モーダルは背景クリック・「閉じる」ボタンの両方で閉じられる。地図ピン→病院シート→募集詳細と進んだ後に「戻る」で押すと病院シートではなく元のタブ一覧に戻る点は経路の厳密な復元ではないが、致命的な迷子は生じないため今回は現状維持（記録のみ・未修正）。
- **モバイル操作性**：主要ボタン・カード・タブは十分なタップ領域（8px以上のパディング）を確保。地図はモバイル時に高さを360pxへ縮小するレスポンシブ対応済み。トーストが長文になった際に画面幅を超えて見切れる可能性があったため、`max-width:88vw`と`word-break`を追加（上記CSS）。
- **用語・トーン**：全画面で「です／ます調」に統一されている。「医籍登録番号」「保険医登録票」等の専門用語は残るが、実在確認という制度上不可欠な語であり、直後に平易な説明文が添えられているため許容範囲と判断（Legal by Design上も必要な情報開示）。
- **Legal by Design**：運営コンソール（`renderAdmin`）を確認したが、医師のレコメンド機能・運営がやりとりに参加する機能は存在しないことを確認した（`sendMessage`もAPI層で運営ロールの送信を拒否）。追加・変更は行っていない。
